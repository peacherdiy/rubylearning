Ruby/Rails 常見問題

問：Rails 好學易上手嗎?

Rails並不算是好學的工具，好學跟好用是不一樣的。Rails是一個針對網站開發一個非常專業的框架。Rails有一個設計哲學是慣例優於設定，為了能夠上手，你必須熟悉這些慣例才能夠加速開發。這就像拿小畫家與Adobe Photoshop相比，前者非常好學，但是如果你要拿來當做吃飯的傢伙，就會選擇Photoshop而不是小畫家，而Photoshop是需要你投資時間下去學的。

對於已經熟悉其他網站開發工具的朋友們，學習第二專長特別困難，因為這可能是你第一次必須先犧牲生產力以換取自我技能的提升。剛開始學習開發速度雖然降低了，但是如果開發網站是你拿來吃飯的專業技能，我相信投資學習Rails最後得到的回報是值得的。

問：Ruby 1.8 還是 Ruby 1.9 好?

目前的用戶數大約是7:3，而 Ruby 1.9 的使用者數會逐漸上升。比較起來，1.8 與套件相容性較佳，有些較舊的套件不一定與 1.9 相容。而 1.9 的執行效能和功能則較為進步，例如 Rails 3.1 已經有一些功能(HTTP Streaming)必須是 Ruby 1.9 才支援，而 Rails 4 更是將不支援 Ruby 1.8。所以，我的建議是如果你沒有已經開發好的 Ruby 1.8 程式包袱，那麼請考慮採用 Ruby 1.9。

問：Why Ruby?

Ruby 是一套非常重視 Usability 的物件導向程式語言，非常看重程式碼的可讀性及維護性。Matz 在設計 Ruby 時，就特別考量一般人容不容易了解(他說我們都是凡人，像Lisp是給神人用的)。這也是為什麼你常常會聽到 Ruby 的程式碼自然簡潔又漂亮。您可以看看這份 Ruby 創造者 Matz 的Why Ruby? 投影片 或是 Matz 在 RubyConf 的演講: 2009、2008，您會更了解及喜愛 Ruby 的哲學。

Ruby 也是目前做 Domain-specific language(DSL)，特別是 Internal DSL 最為成功的程式語言。透過 DSL，程式不但可以擁有非常好的可讀性，也可以大幅增加生產力。

這些例子包括：Rails for Database-backed WebApp、Arel for SQL、Rake for building、Sinatra for WebApp、RSpec for BDD Testing、Chef for configuration management。Cucumber 則是一套成功的 External DSL。這些函式庫正積極地影響我們對軟體開發的想法。我們相信，還會有更多更有趣的 DSL 出現。

問：Why Rails?

這是最常被問到的問題。首先，我要說這是一個從 CGI/Perl -> Template 語言(如PHP、ASP) 到 Web framework 的時代，透過 Web framework 我們不必重新造輪子撰寫基礎功能。更重要的是，程式的可維護性大為增加。熟悉該 framework 的人，可以也快熟悉網站程式的架構。

而各種程式語言要入門上手，其實都不會太困難。我認為重點會在於你能不能夠熟悉做事情的 framework。

所以，撇開程式語言的偏好，Ruby on Rails 是目前 Web framework 中做前端(指生成HTML給瀏覽器) Application Server 最為成功的。它的概念也深深影響了非常多其他程式語言的後進 Web framework，例如 CakePHP、Grails、TurboGears、catalyst 等等(模仿是最大的恭維)。我們可以用非常有效率的程式碼開發出網站應用程式。另外，可能會讓你感到意外的是，它也是目前動態語言中，生態圈最為豐富的框架。

最豐富? 是的，你可以以 Rails 為關鍵字去搜尋書籍、研討會、顧問公司、第三方服務、Plugins 等。例如以 Rails 搜尋 Amazon 電腦類書籍有 60 筆、CakePHP 有 3 筆、Zend Framework 有 12 筆、codeIgniter 有 3 筆、symfony 有 14 筆、Django 有 13 筆、turbogears 有 2 筆。(2010/1)

也有許多特別適合/針對 Rails 的第三方網路服務，例如 Github、Heroko、hoptoad、New Relic、PostageApp等等，都有提供 Ruby 套件支援。

也因為使用 Rails 的人數多，所以在開發上各個方向都有人提供了最佳實務: 從如何寫出好的程式碼、網站安全性、網站性能、擴充性、全文搜尋、非同步處理等等。這是一個非常活躍的社群。

問：我們公司很喜歡 Rails, 去哪裡找 Rails developer?

我必須很誠實的說，目前在台灣的人力市場尋找，例如 104，可能不太容易找到。我的建議是，從有經驗的 PHP/Java/.NET 的網站程式設計師找起，然後透過教育訓練學習 Ruby on Rails, 反而會比較容易。

問： 要學習 Ruby on Rails，目前有什麼中文書籍呢？

因為 Rails 版本進步較快的關係，等到中文翻譯版問世通常過不久版本就過期了(目前到Rails3，書店裡的中文Rails書全部都是Rails2以前版本)，加上譯者都不是專業的 Rails developer，翻對翻錯都不知道，也不會有讀者服務和相容性勘誤。所以你想要找中文書，我只推薦你現在看的這本，不需要再看其他 Rails 中文書。

Ruby 中文書方面，雖然翻譯方面可能差強人意，但是還可以一看。像是 “Ruby 程式設計密技 268, 博碩”, “Ruby Programming－向 Ruby 之父學程式設計, 博碩”, “Ruby 學習手冊, 歐萊禮”, “Ruby Cookbook 錦囊妙技, 歐萊禮”, “Ruby for Rails－Rails 開發者必備的 Ruby 學習手冊, 旗標”, “Ruby 程式設計, 歐萊禮” 等等。市面上的中文書都是 Ruby 1.8 的版本，不過語法幾乎與 1.9 是相容的。Ruby 1.9 與 1.8 最大的明顯差異在於 String Encoding 的處理上。

原文書的選擇就多了，您可以參考Ruby/Rails生態圈所推薦的書籍。

問：Rails 學習資源太少? 人才難找?

回答在 Why Rails? 了，Rails 的學習資源意外的多，人才方面則可以透過教育訓練。好手都很難找，不如挑容易 Training 的 Framework。挑學習力強的人才，而不限定本來一定要會。

問：Rails 佈署會不會很麻煩?

Rails 剛推出時，只能使用 Web server + Application server 的架構，設定上的確比較麻煩。不過在 2008 年推出了 mod_rails (Passenger)，就像 mod_php 一樣可以非常簡單的掛載到 Apache 或 Nginx 中成為一個模組，目前是最為方便的佈署方式，包括 37singals 本身也都使用 Passenger。

如果考慮雲端PaaS服務的話，Heroko是一家做的很成功的廠商。

如果想知道更多，請參考 佈署 一章。

問：Ruby 跟 Rails 的效能會不會很慢?

雖然 Ruby 這個程式語言在演算法效能上並不突出(事實上，任何動態語言都比不過靜態語言啊)，但是你必須了解程式語言的速度並不等於 Framework 的速度。Framework 包含了非常多的 call stack，框架寫的好不好才是關鍵。Rails 在這方面有非常多的關注，很意外地，如果跟 PHP 的框架比起來，Rails 的速度其實還比較快。

問：Rails 可以 Scale 嗎?

Scalability 的能力主要關乎於整體的網站架構，例如分散式 Message Queue、SOA 架構、Database 設計、與第三方服務的API等等，Application server 只是整體架構中的一個環節而已，Rails 的效能並不會是你做擴展上的阻礙(如果你不知道我在說什麼，你的 bottleneck 通常就是資料庫!)。從另一方面來看，Rails 的彈性反而幫助你逐步因應 Scale 而作架構調整。

問：Rails 改版太快跟不上?

只有已經停止開發的函式庫，才不會有更新改版。Rails 目前還是個擁有強大開發能量的框架，而且相較於其他框架，Rails 比較沒這麼保守於向後相容這件事情上，雖然變更的API會上一個版本事先預告，但是版本之間的時間也沒差幾個月啊。伴隨著成長與進步，升級產生的痛似乎是再所難免。也就是說，如果你想一直跟上 Rails 升級的腳步，那麼就必須時常關注 Rails 社群的動向才行。

Rails 版號分成Major、Minor、Tiny，其中Major的改變代表API有很多變更，升級主版號想必是辛苦的。Minor版號則代表新功能以及少部分的API變更，要升級會有點小痛。Tiny版號則為Bug fixes，應該可以無痛升級。如果不想被升級炸到，看到任何一個新版號出來，請盡量忍住不要馬上升級，多等個幾天觀察到到看看有沒有災情，歷史上就曾發生幾次發佈新版本之後，因為發現嚴重問題，隔一天又發佈下一個版本的糟糕記錄(2.0.0->2.0.1只隔了一天, 2.3.6->2.3.7->2.3.8都只隔了一天，3.0.8->3.0.9隔了一天)。升級前做好功課，了解有哪些變更、有沒有哪些Gem套件有問題、佈署到 staging server 做測試，這些都是要做的。

另一方面要注意的事情，就是平常開發就注意減低升級上的困難。如果你使用的是 Rails 核心 API，這些在升級文件中都會告訴你應該注意的事項。因此有困難的多半是 Gems和Plugin 的關係。 因此在使用 Plugin 上，我建議你 應該把焦點集中於你正在使用中的 plugin 的維護及更新狀況? 有很多人用嗎? 有人維護的套件，在 rails 發佈新版之後，就會有新的對應版本。另外，你熟悉這個套件嗎? 你看的懂這個 plugin 到底在做什麼事情嗎? 很多時候，我們其實不太需要太過 hack 的 plugin，用 monkey patch 的套件容易有問題、來路不明的 Plugin 也少碰為妙。減少不必要的 gem dependency 可以減低升級上的困難。
